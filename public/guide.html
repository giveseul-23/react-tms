📘 DEVELOPER GUIDE

화면 제작 가이드

이 프로젝트의 모든 업무 화면은 MVC 5-파일 패턴 + MasterDetailPage를 기반으로 제작합니다.
새 화면을 추가하거나 레이아웃을 변경할 때 이 문서를 참고하세요.

🗂 전체 구조

모든 업무 화면은 아래 5개 파일로 구성됩니다. 각 파일은 단일 책임 원칙에 따라 역할이 분리됩니다.

View
{화면명}.tsx

UI 렌더링만 담당. MasterDetailPage로 조립. 비즈니스 로직 없음.

Controller
{화면명}Controller.tsx

이벤트 핸들러 · API 호출. 액션 정의(useCallback).

Model
{화면명}Model.ts

화면 상태(useState/useRef) + 공통코드 codeMap.

Columns
{화면명}Columns.tsx

그리드 columnDefs. LBL_* 다국어 키 사용.

Api
{화면명}Api.ts

apiClient.post 호출. MENU_CD + withSession 자동.

folder structure
src/features/{모듈}/{화면}/
├── {화면}.tsx               // View       — UI 렌더링만
├── {화면}Controller.tsx     // Controller — 이벤트·API 호출
├── {화면}Model.ts           // Model      — 상태 관리
├── {화면}Columns.tsx        // Columns    — 그리드 컬럼

🌊 컴포넌트 & 데이터 흐름

5개 파일이 어떻게 연결되는지, props 가 어디서 어디로 흐르는지, 조회 한 번에 무슨 일이 일어나는지 한 눈에 보는 도식입니다.

A. 컴포넌트 트리 & props 흐름

View 가 컴포지션 루트. model 객체 하나가 SearchFilters / Controller / DataGrid 셋을 잇는 다리입니다.

컴포넌트 트리
// {화면}.tsx (View) — 컴포지션 루트
┌──────────────────────────────────────────────────────────────────────────┐
│ export default function Feature() {                                       │
│   const model = useFeatureModel(MENU_CODE);      // ① Model — 상태/refs   │
│   const ctrl  = useFeatureController({ model });    // ② Controller — 핸들러  │
│                                                                            │
│   return (                                                                 │
│     <MasterDetailPage menuCode={MENU_CODE}>          // ③ Preset (레이아웃)    │
│       │                                                                    │
│       ├─ useResolvedSearchMeta(menuCode)             // preset 내부 자동       │
│       │     └─ SearchMeta 로딩 + Skeleton 자동                            │
│       │                                                                    │
│       ├─ <SearchFilters                                                       │
│       │     fetchFn={ctrl.fetchList}                // ← Controller          │
│       │     onSearch={ctrl.onSearchCallback}             // ← Controller          │
│       │     searchRef={model.searchRef}              // ← Model (ref 다리)     │
│       │     filtersRef={model.filtersRef}            // ← Model                │
│       │     pageSize={model.pageSize}                // ← Model                │
│       │  />                                                                │
│       │     └─ 내부에서 useSearchExecute             // handleSearch 생성    │
│       │         └─ searchRef.current = handleSearch    // 외부 트리거 통로     │
│       │                                                                    │
│       ├─ master={ <DataGrid {...model.bind("main")}    // rowData/total/page 자동│
│       │              columnDefs={MAIN_COLUMN_DEFS}     // ← Columns             │
│       │              codeMap={model.codeMap}           // ← Model (공통코드)     │
│       │              onRowClicked={ctrl.onMainGridClick} // ← Controller       │
│       │              actions={ctrl.mainActions}        // ← Controller          │
│       │           /> }                                                     │
│       │                                                                    │
│       └─ detail={ <DataGrid {...model.bind("detail")} ... /> }              │
│     </MasterDetailPage>                                                          │
│   );                                                                       │
│ }                                                                          │
└──────────────────────────────────────────────────────────────────────────┘
💡
핵심: model 객체에 모든 ref / state / setData 가 모여 있고, ctrl 객체에 모든 핸들러가 모여 있습니다. 같은 model 인스턴스가 SearchFilters / DataGrid / Controller 셋에 동시에 전달되므로, 셋이 동일한 ref(searchRef, filtersRef, grids.main 등)를 공유합니다.

"조회" 버튼 클릭 → 화면에 결과 표시까지 1초 동안 일어나는 일을 시간 순서대로 보면:

조회 흐름 (사용자 클릭 / Controller 트리거 2가지 진입점)

┌──────────────────────────────────────────────────────▼──────────┐
│ SearchFilters 내부 handleSearch(page) 실행                        │
│   (useSearchExecute 가 만든 함수, model.searchRef.current 에 꽂힘) │
└────────────────────────────┬──────────────────────────────────────┘
                             │ params 조립 (검색조건 + page + limit)

│ onSearch(data) = ctrl.onSearchCallback(data)                          │
│   ├─ model.grids.main.setData(data)        // 그리드 데이터 set     │
│   └─ ctrl.onMainGridClick(data.rows[0])     // 첫 행 자동 선택      │

⚠️
검색 콜백이 두 단계로 나뉘는데 서로 다른 함수입니다.
  • SearchFilters 내부의 handleSearch(page) — 실행자 (fetch 수행)
  • Controller 의 ctrl.onSearchCallback(data) — 수신자 (결과 받아서 grid에 set)
전자가 끝나면 후자가 onSearch 콜백으로 호출되는 직렬 연결입니다.

C. 핵심 ref 다리 — searchRef / filtersRef / rawFiltersRef

컴포넌트 트리 바깥(Controller / 액션 콜백)에서 SearchFilters 내부 함수/상태를 호출하기 위해 ref 3개로 통신합니다. 모두 useBaseModel 또는 View 가 만들고, SearchFilters / Controller 가 공유합니다.

ref생성 위치역할 / 들어있는 값주 사용자
model.searchRef useBaseModel 실행 트리거 통로. SearchFilters 내부 useSearchExecute 가 searchRef.current = handleSearch 로 매 deps 변경마다 최신 함수 꽂음. Controller 가 호출해서 메인 조회 재실행. Controller (base.search() 가 내부적으로 호출)
model.filtersRef useBaseModel 마지막 조회 시 쓰인 검색 params 객체 보관 (페이지/limit 제외). 액션 / 엑셀 다운로드에서 현재 검색 조건 재사용 시 참조. Controller — 액션 콜백, makeExcelGroupAction.fetchFn
rawFiltersRef View 의 useRef (필요 화면만) SRCH_* 접두 raw 조건 맵. POPUP / 동적 컬럼 같이 검색조건의 개별 값을 키 단위로 꺼내야 할 때 사용. 예: srchObj.SRCH_AP_DIV_CD Controller — fetchList 안 동적 컬럼 fetch (ApDailyManagement 등)
💡
왜 ref 인가: handleSearch 가 사용하는 searchState / meta 는 SearchFilters 내부 state. Controller 가 직접 접근할 수 없으므로, ref 슬롯에 함수를 꽂아 통신합니다. forwardRef + useImperativeHandle 의 ref 버전이라 보시면 됩니다.

D. hook 호출 시점 요약

화면이 마운트되면 hook 들이 아래 순서로 한 번씩 실행됩니다.

hook 호출 순서
// View 가 마운트되는 순간
1. useFeatureModel(MENU_CODE)
   └─ useBaseModel<K>(menuCode, opts?)
        ├─ searchRef / filtersRef = useRef(null)
        ├─ grids = Proxy (gridKey 첫 접근 시 lazy 등록)
        ├─ pageSize / layout state
        └─ storageKeys = { outer, top, bottom, layout }  // menuCode → kebab-case
   └─ useCommonStores({ ...storeMap }) → codeMap 빌드

2. useFeatureController({ model })
   └─ useBaseController<K>({ model })
        ├─ callAjax / alert / confirm
        ├─ search / searchSub / resetGrids
        ├─ handleRowClick / addRow / requireParentRow
        └─ saveGrid (dirty 추출 + dsSave 자동)
   └─ useCallback 으로 핸들러 정의 + useMemo 로 actions 배열

3. <MasterDetailPage menuCode={...}> 렌더
   └─ useResolvedSearchMeta(menuCode, meta?)   // preset 내부 자동
        ├─ useSearchMeta(menuCode) 호출 → meta 로딩
        └─ 로딩 중이면 Skeleton 반환 (gate)

4. preset 내부 <SearchFilters> 마운트
   └─ useSearchState / useModuleDefault / useSearchExecute
        └─ useEffect 로 searchRef.current = handleSearch 꽂음

E. storageKey 흐름 — SplitPane 영속화 키

useBaseModel 이 menuCode 로 4개 키를 자동 생성해서 model.storageKeys 에 담아둡니다. View 는 그 중 필요한 것을 골라 preset / SplitPane 에 전달합니다.

F. useCallback / useMemo / deps

Controller 안에서 핸들러는 useCallback, 액션 배열은 useMemo 로 감싸는 이유와 deps 배열의 의미입니다.

한 줄 기준

만들려는 것사용
함수useCallback(fn, deps)
객체 / 배열 / 계산 결과 값useMemo(() => value, deps)

사실 useCallback(fn, deps) 는 useMemo(() => fn, deps) 의 단축 표현. 내부 동작 동일.

왜 필요한가

컴포넌트가 다시 렌더될 때마다 함수 본문 전체가 다시 실행됩니다. 그러면 const x = () => ... 처럼 만든 것도 매번 새 인스턴스로 만들어져요. useCallback / useMemo 가 "deps 가 안 바뀌었으면 지난 번 인스턴스 그대로 재사용" 해서 자식의 불필요한 재렌더를 막아줍니다.

deps 가 뭐냐

두 번째 인자로 들어가는 의존성 배열. "이 안에 든 값들이 바뀌면 다시 만들고, 안 바뀌었으면 지난 번 거 그대로 써" 라는 의미. 함수 본문에서 참조하는 외부 변수를 모두 적어야 합니다.

deps 형태의미
[] (빈 배열)평생 한 번만 만들어 그대로 유지
[base]base 가 새 인스턴스가 되면 다시 만듦
[base, model]둘 중 하나라도 바뀌면 다시 만듦
(생략)매 렌더마다 새로 만듦 — useCallback 쓰는 의미 없음
Controller 패턴 예 — 함수는 useCallback, 배열은 useMemo
// 핸들러(함수) → useCallback
const onMainGridClick = useCallback(
  (row) => base.handleRowClick("main", row, [...]),
  [base],                                // ← deps
);

const onSaveMain = useCallback(
  () => base.saveGrid("main", api.save),
  [base],
);

// 액션 배열 → useMemo
const mainActions: ActionItem[] = useMemo(
  () => [
    makeAddAction({ onClick: onAddMain }),
    makeSaveAction({ onClick: onSaveMain }),
    makeExcelGroupAction({ ... }),
  ],
  [onAddMain, onSaveMain, model],        // ← 안의 핸들러/model 바뀌면 새 배열
);

// 무거운 계산 → useMemo
const codeMap = useMemo(() => {
  const map = {};
  Object.entries(stores).forEach(...);   // 비싼 변환
  return map;
}, [stores]);
💡
ESLint 의 react-hooks/exhaustive-deps 룰이 deps 누락을 자동으로 검사합니다. 빨간 줄이 그어지면 그 변수를 deps 에 추가하세요. 누락하면 stale closure (낡은 값 가둠) 버그 발생 가능.

G. model vs base — 헷갈리지 않기

둘은 비슷해 보이지만 완전히 별개의 객체입니다. base 가 model 안에 들어있는 게 아니에요.

modelbase
만든 hookuseBaseModeluseBaseController
역할데이터 보관소헬퍼 도구상자
안에 든 것state, refs, setData, grids, storageKeys, codeMapcallAjax, search, saveGrid, handleRowClick, addRow 등 함수
관계base 의 인자로 넘겨줌받은 model 의 값을 읽고 씀
어디서 살아있나View / Controller 둘 다 접근Controller 안에서만 (로컬 변수)
둘의 관계 — 코드로 보기
// View 안
const model = useFeatureModel(MENU_CODE);       // ← Model hook 이 만든 객체
const ctrl  = useFeatureController({ model });
//                                  ^^^^^^^
//                                  Controller 가 model 을 인자로 받음

// Controller 안
function useFeatureController({ model }) {
  const base = useBaseController({ model });
//                                  ^^^^^^^^
//                                  base 가 model 을 참조

  // base 의 헬퍼들이 내부에서 model 의 값을 읽고 씀
  const onSave = useCallback(() => {
    base.saveGrid("main", api.save);
    //   └─ 내부 동작: model.grids.main.ref.current.rows 읽고
    //                  → api 호출 → model.searchRef.current?.() 호출
  }, [base]);

  // 주의: base 자체는 return 에 안 넣음 (Controller 의 로컬 변수)
  return { fetchList, onSearchCallback, mainActions, ... };
}

비유

  • model = 부엌 (창고/재료/도구가 다 들어있는 곳)
  • base = 요리사 (부엌 재료를 가지고 일을 처리하는 헬퍼)
  • 요리사는 부엌이 있어야 일할 수 있지만, 요리사가 부엌 안에 들어있는 건 아님
💡
왜 헷갈리나: base 의 모든 헬퍼가 model 의 state 를 건드리니까 마치 model 의 일부 같지만, base 는 별도의 객체입니다. useBaseController 가 인자로 받은 model 을 closure 로 캡처해서 헬퍼들이 쓰는 구조예요.

H. 검색조건 state 흐름 — useSearchState / useModuleDefault

SearchFilters 내부의 두 hook. 검색조건의 "지금 값"을 누가 어떻게 갱신하는지와, 로그인 사용자의 디폴트 값이 어디서 자동으로 채워지는지를 설명합니다.

한 줄 요약

Hook역할
useSearchState검색조건 state CRUD (입력 / 초기화 / 단일 필드 set)
useModuleDefault외부 모듈에서 사용자 디폴트 값을 자동으로 가져와서 채워줌 (예: 본인 소속 사업부/물류그룹)

useSearchState 반환값

반환용도
searchState현재 검색조건 객체 (사용자 입력 누적)
setSearchState직접 갱신 (useState 의 setter)
getCondition(key)특정 필드의 현재 값 조회
updateCondition(key, value, op, type, source?)사용자가 한 글자 칠 때마다 호출 — 필드 하나만 갱신
handleReset"초기화" 버튼 클릭 시 — 기본값 + 모듈 디폴트 캐시 재적용
buildInitialSearchState메타로부터 초기 state 빌드 (YMD/YMDT 만 오늘 채움)

updateCondition 한 글자 칠 때 일어나는 사이클

사용자가 input 한 글자 칠 때 어떤 흐름으로 state 가 갱신되는지:

state 갱신 사이클
사용자가 input 에 "a" 입력
   │
   ▼
SearchFieldRenderer 의 onChange 발화
   │  updateCondition("LGST_GRP_CD", "a", "equal", "STRING")
   ▼
useSearchState 의 updateCondition 실행
   │  setSearchState((prev) => ({ ...prev, LGST_GRP_CD: { value: "a", ... } }))
   ▼
[★ React 가 처리 ★]
   │  1. React 내부 state store 에 새 값 저장
   │  2. SearchFilters 를 재렌더 큐에 등록
   │  3. 다음 렌더 사이클 트리거
   ▼
SearchFilters 재렌더
   │
   ├─ searchState 변수가 새 값을 가리킴
   │
   ├─ useSearchExecute 다시 실행
   │     ↑ handleSearch = useCallback(..., [searchState, ...])
   │     ↑ deps 바뀜 → handleSearch 새 함수로 재생성
   │
   ├─ useEffect([searchRef, handleSearch]) 실행
   │     ↑ searchRef.current = handleSearch  // 최신 함수 꽂기
   │
   └─ SearchFieldRenderer 재렌더 → input value 새 값 반영
⚠️
중요: updateCondition 은 state 를 직접 안 바꿉니다. setSearchState 를 호출해 React 에게 "이 값으로 바꿔달라"고 요청할 뿐. 실제 갱신은 React 가 처리(state 슬롯 업데이트 + 재렌더). 직접 searchState[key] = ... 대입하면 React 는 변경을 감지 못 함.
💡
매 글자마다 fetch 가 안 일어나는 이유: searchState 가 바뀌면 handleSearch 가 재생성되고 ref 에 새로 꽂히지만, 함수 자체가 실행되진 않습니다. "다음에 누군가 호출하면 최신 조건으로 실행할 준비" 상태가 될 뿐. 실제 fetch 는 [조회] 버튼 클릭 또는 Controller 가 base.search() 호출할 때만.

useModuleDefault 자동 디폴트 채움

View 에서 searchProps.moduleDefault: "TMS" prop 을 넘기면 활성화. 로그인 사용자의 디폴트 값(예: 본인 소속 사업부 DIV_CD, 물류그룹 LGST_GRP_CD)을 자동 조회해서 검색조건에 채워줍니다 (센차 setModuleDefaultValue 대응).

useModuleDefault 흐름
1. 메타 로드됨 (loadedRef = false, 첫 실행만)
   ▼
2. commonApi.fetchModuleDefaultValue("TMS", apiParams)
   ▼ 응답: { LGST_GRP_CD: "L001^SPLT^본사물류그룹", DIV_CD: "D001^SPLT^본부", ... }
   ▼
3. 응답을 "^SPLT^" 로 split → code / name 분리
   ▼
4. POPUP 타입이면 _CD / _NM 두 키로 분해
   defaults["LGST_GRP_CD"] = "L001"
   defaults["LGST_GRP_NM"] = "본사물류그룹"
   ▼
5. moduleDefaultCacheRef.current = defaults   // reset 재사용용 캐시
   ▼
6. setSearchState((prev) => merge(prev, defaults))   // 화면에 반영

두 hook 의 협력 — 공유 ref 다리

moduleDefaultCacheRef 는 useSearchState 와 useModuleDefault 가 공유하는 ref. 두 hook 의 통신 다리입니다.

동작주체
API 응답을 캐시에 저장 (한 번만)useModuleDefault
"초기화" 클릭 시 캐시 읽어 재적용useSearchState.handleReset

캐시 없으면 매 초기화마다 API 재호출 — 캐시로 한 번만 호출하고 영속 보관.

시간 순 흐름

SearchFilters 마운트 → 사용자 입력 → 초기화 클릭
[T+0] 컴포넌트 마운트
       ├─ useSearchState: useState({}) → searchState = {}
       ├─ useModuleDefault: useEffect 등록 (meta 없어서 대기)
       └─ moduleDefaultCacheRef: { current: null }

[T+1] 메타 로드 완료 (useResolvedSearchMeta)
       │
       ├─ useSearchState 의 useEffect 발화
       │   └─ buildInitialSearchState() → 날짜 필드만 오늘로
       │   └─ setSearchState(initial)
       │
       └─ useModuleDefault 의 useEffect 발화 (loadedRef=false)
           └─ API 호출 → 응답
           └─ moduleDefaultCacheRef.current = defaults
           └─ setSearchState((prev) => merge(prev, defaults))

[T+2] 사용자 타이핑 (매 입력마다)
       └─ updateCondition(key, value, ...) → searchState 갱신

[T+3] 사용자 "초기화" 클릭
       └─ handleReset()
           └─ buildInitialSearchState() (날짜 오늘)
           └─ moduleDefaultCacheRef 캐시 머지 (API 재호출 X)

import { apiClient } from "@/app/http/client";
import { getSessionFields } from "@/app/services/auth/auth";

const MENU_CD = "MENU_XXX";

const withSession = (payload = {}) => {
  const s = getSessionFields();
  return Array.isArray(payload)
    ? payload.map((it) => ({ ...s, ...it }))
    : { ...s, ...payload };
};

export const featureApi = {
  MENU_CD,
  getList(payload) {
    return apiClient.post("/featureService/searchList",
      withSession({ MENU_CD, ...payload }));
  },
  save(payload) {
    return apiClient.post("/featureService/save",
      withSession(payload.dsSave));
  },
};
