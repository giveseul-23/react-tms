1. 개요

DataGrid 는 ag-grid-community 위에 우리 프로젝트 컨벤션을 입혀 둔 표준 그리드 컴포넌트입니다. 대부분의 화면은 useBaseModel 의 model.bind(gridKey) 와 함께 사용하며, 그리드별 rowData / selected / 페이지 / setter 가 자동으로 spread 됩니다.

  • 한 화면에서 여러 개의 그리드(main / sub01 / sub02 …)를 selection 기반 cascade 로 묶을 수 있도록 설계되어 있습니다.
  • 공통 동작 (audit 컬럼 자동, EDIT_STS 기반 편집 잠금, 공통코드 → 라벨 자동 치환, autoSize, 페이지네이션) 이 내장됩니다.
  • View 레이어에서는 JSX/레이아웃만 작성하고, 핸들러는 Controller, 상태는 Model 로 위임합니다.
TIP. 신규 화면은 src/features/guide/Guide_Feature* 5종 세트를 복사하면 모든 권장 패턴이 이미 반영되어 있습니다.

2. 기본 사용법 — {...model.bind("main")}

View 에서 DataGrid 를 쓸 때, 일반 prop 을 일일이 잇지 않고 model.bind() 의 결과를 spread 합니다. 다음 prop 들이 자동으로 채워집니다.

  • rowData · setRowData
  • totalCount · currentPage · pageSize
  • onPageChange · onPageSizeChange
  • selectedRow (controller 가 setSelected 로 박은 행을 시각 반영)
  • onSelectionChanged (사용자 클릭으로 변경된 selection 을 model 에 동기화)
  • audit: true (auto-audit 컬럼)
// View — 가장 기본형
<DataGrid
  {...model.bind("main")}
  columnDefs={MAIN_COLUMN_DEFS}
  onRowClicked={ctrl.onMainGridClick}
  actions={ctrl.mainActions}
/>
주의. View 에서 useRef 로 searchRef / filtersRef 를 별도로 만들지 마세요. useBaseModel 이 이미 들고 있으므로 model.searchRef / model.filtersRef 를 사용해야 합니다.

3. Columns 규칙

XxxColumns.tsx 는 단순 const 배열로 정의합니다. 다음 컨벤션이 자동으로 처리됩니다.

3-1. 자동 변환 룰

조건자동 동작
headerName: "LBL_*"다국어 키로 인식 — Lang.get 자동 적용
field 가 DTTM 포함자동 날짜 포맷
field 가 _STS 로 끝남자동 중앙 정렬
type: "numeric" 또는 dataType: "number"자동 우측 정렬
headerName: "No"자동 일련번호 + 고정 너비
codeKey: "xxxTcd"codeMap[xxxTcd][value] 로 라벨 자동 치환 (cellRenderer 자동 주입)

3-2. 편집 가능 여부 — EDIT_STS 기반 자동 변환

ag-grid 의 editable 을 직접 쓰지 않고, insertable / editable 키워드로 의도를 선언합니다. 실제 ag-grid 의 editable 은 EDIT_STS 와 조합되어 자동 산출됩니다.

선언의미
insertable: true추가 상태 행 (EDIT_STS === "I") 에서만 편집 가능
editable: true수정 상태 행 (EDIT_STS !== "I") 에서만 편집 가능
insertable: true, editable: true항상 편집 가능
둘 다 미지정 / false편집 불가
editable: (params) => ...함수형은 ag-grid 에 그대로 위임

3-3. PK 컬럼

  • isPrimaryKey: true 표시 → DataGrid 가 첫행 자동선택을 자동 활성화 (View 에서 별도 prop 명시 불필요). 키 컬럼은 자동으로 PK 매칭에 사용됨.
  • 보통 isPrimaryKey: true + insertable: true 조합으로 사용 — 추가 시 입력, 수정 시 잠금.

3-4. Columns 정의 예시

export const MAIN_COLUMN_DEFS: ColDef[] = [
  { headerName: "No" },                                  // 자동 일련번호
  { headerName: "LBL_VEHICLE_NO", field: "VEH_NO",
    isPrimaryKey: true, insertable: true },           // PK + 추가 입력
  { headerName: "LBL_VEHICLE_TYPE", field: "VEH_TCD",
    codeKey: "vehTcd", editable: true },          // 코드→라벨 + 수정
  { headerName: "LBL_AMT", field: "AMT",
    type: "numeric", editable: true },                // 우측정렬
  { headerName: "LBL_REG_DTTM", field: "REG_DTTM" },     // 날짜 자동포맷
];

4. Audit 컬럼 자동 추가

등록자/등록일/수정자/수정일/행상태/삭제 같은 표준 audit 컬럼은 컬럼 배열에 직접 넣지 않습니다. model.bind() 가 자동으로 audit: true 를 spread 하면 DataGrid 가 컬럼 끝에 자동 추가합니다.

audit 값동작
true (또는 미지정 + bind)standardAudit 모두 ON
false자동 추가 안 함 (audit 컬럼을 직접 중간 위치에 둘 때)
{ updatePerson: false, ... }부분 토글
 
// audit 컬럼 일부만 끄기
<DataGrid
  {...model.bind("main")}
  audit={{ updatePerson: false }}
  columnDefs={MAIN_COLUMN_DEFS}
/>

// 완전히 끄기
<DataGrid
  {...model.bind("sub01")}
  audit={false}
  columnDefs={SUB_COLUMN_DEFS}
/>

6. 레이아웃 — plain / tab / render 자유 조립

6-1. plain (단일 그리드)

<DataGrid
  {...model.bind("main")}
  layoutType={"plain"}
  columnDefs={MAIN_COLUMN_DEFS}
  actions={ctrl.mainActions}
/>

6-2. tab — presets[key].render 패턴 (권장)

탭마다 단순히 컬럼/액션만 바꾸는 경우라도 실무에선 render: () => (...) 로 통일해 사용합니다. 그래야 6-3 의 자유 조립과 자연스럽게 이어집니다. 

<DataGrid
  layoutType={"tab"}
  tabs={[
    { key: "AP", label: "LBL_AP" },
    { key: "AR", label: "LBL_AR" },
  ]}
  presets={{
    AP: {
      render: () => (
        <DataGrid
          {...model.bind("ap")}
          columnDefs={AP_COLS}
          actions={ctrl.apActions}
        />
      ),
    },
    AR: {
      render: () => (
        <DataGrid
          {...model.bind("ar")}
          columnDefs={AR_COLS}
          actions={ctrl.arActions}
        />
      ),
    },
  }}
  onTabChange={ctrl.onTabChange}
  actions={[]}
/>

6-3. tab + render 자유 조립 (SplitPane / 중첩 DataGrid / 중첩 탭)

render 안에서는 무엇이든 자유롭게 조립할 수 있습니다. SplitPane 으로 좌/우 패널을 만들고, 우측에 또 다른 탭형 DataGrid 를 중첩할 수도 있습니다. ApSettlMgmt 가 이 패턴의 대표 사례입니다. 

// SUMMARY 탭만 좌(요약 그리드) + 우(중첩 4 sub-tab) 좌우 분할
<DataGrid
  layoutType={"tab"}
  tabs={[
    { key: "SUMMARY", label: "LBL_SUMMARY_LIST" },
    { key: "COST_CENTER", label: "LBL_EACH_COST_ACCOUNT" },
  ]}
  presets={{
    SUMMARY: {
      render: () => (
        <SplitPane direction={"horizontal"} defaultSizes={[30, 70]}>
          <DataGrid
            {...model.bind("summary")}
            columnDefs={SUMMARY_COLS}
            actions={ctrl.summaryActions}
          />
          <DataGrid
            layoutType={"tab"}
            tabs={[
              { key: "MONTHLY_FARE", label: "LBL_FIXED_VEH_CHARGE" },
              { key: "FREIGHT",      label: "LBL_ITM_QTY_RATE" },
            ]}
            presets={{
              MONTHLY_FARE: {
                render: () => (
                  <DataGrid {...model.bind("monthlyFare")} columnDefs={MONTHLY_FARE_COLS} />
                ),
              },
              FREIGHT: {
                render: () => (
                  <DataGrid {...model.bind("freightPay")} columnDefs={FREIGHT_COLS} />
                ),
              },
            }}
            actions={[]}
          />
        </SplitPane>
      ),
    },
    COST_CENTER: {
      render: () => (
        <DataGrid
          {...model.bind("costCenter")}
          columnDefs={COST_CENTER_COLS}
          actions={ctrl.costCenterActions}
        />
      ),
    },
  }}
  actions={[]}
/>
왜 render 패턴인가. 탭 단위로 컴포넌트 트리 자체를 결정할 수 있어서 — SplitPane, 중첩 탭, 폼 패널 같은 복합 레이아웃을 외부 분기 없이 한 곳에 응집해 표현할 수 있습니다.

7. Tree Data

계층 데이터는 treeData 모드를 사용합니다. 각 행에서 트리 경로를 만들어 주는 getDataPath 가 필수. 

<DataGrid
  {...model.bind("main")}
  treeData
  getDataPath={(row) => row.PATH.split("/")}
  groupDefaultExpanded={-1}     // -1: 전체 펼침, 0: 전체 접힘, N: N 레벨까지
  autoGroupColumnDef={{
    headerName: "LBL_NAME",
    cellRendererParams: { suppressCount: true },
  }}
  columnDefs={TREE_COLUMN_DEFS}
/>

별도로 TreeGrid 컴포넌트 (src/app/components/grid/TreeGrid/) 도 있어 메뉴/조직 같이 전형적인 트리 화면에서 사용할 수 있습니다.

8. 페이지네이션

페이지네이션은 pagination=true(기본) + totalCount 가 있을 때만 표시됩니다. model.bind 가 totalCount / currentPage / pageSize / onPageChange / onPageSizeChange 를 자동으로 묶어주므로, View 에서는 별도로 신경 쓸 필요가 없습니다.

  • 기본 pageSize 는 500.
  • onPageChange 시 base.search(page) 를 호출하면 메인 조회가 다시 돕니다.
  • 페이지네이션을 끄려면 pagination={`false`}.

9. 선택 / Cascade

Master → Detail 같은 cascade 흐름은 Controller 에서 base.handleRowClick 헬퍼로 표현합니다. 

// Controller
const onMainGridClick = (row: MainRow) => {
  base.handleRowClick("main", row, {
    // row 가 바뀌면 sub01 을 비우고 새로 fetch
    sub01: { fetch: () => api.getSub01({ VEH_NO: row.VEH_NO }) },
  });
};

// View — onRowClicked 만 잇기만 하면 됨
<DataGrid
  {...model.bind("main")}
  columnDefs={MAIN_COLUMN_DEFS}
  onRowClicked={ctrl.onMainGridClick}
/>
  • selection 시각 동기화: selectedRow 는 controller 가 setSelected 로 박은 행을 시각으로 반영. (자동 spread)
  • 첫행 자동선택: columnDefs 안에 isPrimaryKey: true 가 1개 이상 있고 onRowClicked 가 있으면 자동 활성화. rowData 갱신 시 이전 선택을 PK 로 매칭해 복구하고, 매칭 실패 시 아무 동작도 하지 않습니다.

10. 전체 예제 (요약)

Guide_Feature 패턴을 따른 가장 표준적인 화면 한 묶음의 핵심만 압축한 예시입니다.

Xxx.tsx (View)

export const MENU_CODE = "XXX001";

export default function Xxx() {
  const model = useXxxModel(MENU_CODE);
  const ctrl  = useXxxController(model);

  return (
    <MasterDetailPage
      menuCode={MENU_CODE}
      searchProps={{ /* meta 기반 자동 */ }}
      direction={model.layout === "side" ? "horizontal" : "vertical"}
      layoutToggle={{ layout: model.layout, onToggle: model.setLayout }}
      storageKey={model.storageKeys.outer}
      top={
        <DataGrid
          {...model.bind("main")}
          columnDefs={MAIN_COLUMN_DEFS}
          onRowClicked={ctrl.onMainGridClick}
          actions={ctrl.mainActions}
        />
      }
      bottom={
        <DataGrid
          {...model.bind("sub01")}
          columnDefs={SUB01_COLUMN_DEFS}
          actions={ctrl.sub01Actions}
        />
      }
    />
  );
}

XxxController.tsx (핸들러)

const base = useBaseController<GridKey>({ model });

const onMainGridClick = (row: MainRow) => {
  base.handleRowClick("main", row, {
    sub01: { fetch: () => api.getSub01({ VEH_NO: row.VEH_NO }) },
  });
};

const mainActions = useMemo(() => makeCommonActions({
  add:  () => base.addRow("main", EMPTY_MAIN),
  save: () => base.saveGrid("main", api.save, { afterSave: "refresh" }),
}), [base]);

11. 내부 구조 — 파일 분리 & Hook

DataGrid 는 본체(index.tsx)에서 props 분배·레이아웃·JSX 만 책임지고, 실제 로직은 책임 단위로 hook 과 유틸 파일로 잘라 두었습니다. 본체는 hook 들이 도출한 값을 조립해 <AgGridReact /> 에 넘기는 오케스트레이터 역할만 합니다.

11-1. 폴더 트리

DataGrid/
├─ index.tsx                  # 본체 (오케스트레이터)
├─ types.ts                   # GridTab, GridPreset 타입
├─ autosize.ts                # canvas 텍스트 측정 유틸 (순수 함수)
├─ Pagination.tsx             # 하단 페이지네이션 UI
├─ GridTabs.tsx               # 상단 탭 UI
├─ TrackPanel.tsx             # 하단 추적 결과 expand 패널
├─ useActivePreset.ts         # tab/preset 활성화 분기 + audit + 컬럼 변환
├─ useAutoSize.ts             # 컬럼 너비 자동 계산 트리거
├─ useRowLifecycle.ts         # selection / rowData effect 묶음 (4가지)
├─ useGridHandlers.ts         # ag-grid 이벤트 → 외부 콜백 라우팅
├─ useGridProps.ts            # AgGridReact 에 spread 할 prop 메모
└─ useCellRangeSelection.ts   # 셀 드래그 범위 선택 + Ctrl+C TSV

11-2. 데이터 흐름

본체 index.tsx 는 단순히 다음 체인을 조립합니다. 각 hook 은 위 단계의 결과를 입력으로 받아 자신의 출력을 만듭니다. 

props (rowData, columnDefs, audit, codeMap, tabs, presets, ...)
  │
  ▼
useActivePreset   ── tab 활성 시 presets[activeTab] 우선 / audit spread / processColumnDef
  │          → activeColumnDefs · activeRowData · summaryRow · activeActions
  │          → activeOnRowClicked · activeOnCellValueChanged · activeRender
  │          → effectiveRowKeys · effectiveAutoSelect · finalColumnDefs
  ▼
useAutoSize       ── onGridReady · onFirstDataRendered · autoSizeKey effect
useRowLifecycle   ── 4 effects (선택 sync · PK 복원 · 신규행 스크롤 · check default)
useGridHandlers   ── handleRowClicked · handleRowSelected · handleSelectionChanged ...
  │
  ▼
useGridProps      ── commonGridProps (ref 안정화: cellEditor 생존)
  │
  ▼
<AgGridReact {...commonGridProps} rowData={activeRowData} />

useCellRangeSelection   ── 컨테이너 ref 에 직접 listener 부착 (병렬)

tab 모드일 때 presets[activeTab] 의 값을 우선 적용해 active* 시리즈를 메모이즈. 추가로 audit 컬럼 자동 spread, processColumnDef 변환(Lang/align/type/DTTM/numeric/_STS/codeKey/insertable/editable/isPrimaryKey), summable 합계 row 생성, isPrimaryKey → effectiveRowKeys 자동 추출까지 처리.
핵심 입력
layoutType, activeTab, presets, columnDefs, rowData, audit, codeMap, setRowData, onRowClicked
핵심 출력
activeColumnDefs · activeRowData · summaryRow · activeActions · activeOnRowClicked · activeOnCellValueChanged · activeCodeMap · activeRender · effectiveRowKeys · effectiveAutoSelect · finalColumnDefs
왜 분리
DataGrid 의 분기 복잡도 중 가장 큰 부분 — "탭/프리셋이 있냐 없냐" 가 모든 active* 값을 흔드는데, 이걸 한곳에 모아 두면 다른 hook 들은 active* 만 받아 단순하게 동작할 수 있음

useAutoSize.ts — 컬럼 너비 자동 계산

역할
onGridReady / onFirstDataRendered 시점에 autosize.ts 의 측정 유틸로 모든 컬럼의 최적 너비를 계산해 ag-grid api 에 적용. 이후로는 autoSizeKey 가 변할 때만 재실행.
핵심 입력
disableAutoSize, finalColumnDefs, activeRowData, activeTab, autoSizeKey, gridApiRef, columnOrderRef
핵심 출력
handleGridReady · handleFirstDataRendered (콜백)
왜 분리
dep 으로 activeRowData 가 아니라 autoSizeKey 를 사용 — model.bind 가 객체 set(조회 결과 도착) 에만 +1 해 주므로, 셀 편집/행 추가 같은 함수형 setter 호출에는 autoSize 가 재발화하지 않아 편집 중 가로 스크롤이 유지됨. 이 미묘한 dep 룰을 hook 안에 가둬 둠.

역할
① controller 가 setSelected 로 박은 selectedRow 를 ag-grid 시각 선택으로 반영
② effectiveAutoSelect 일 때 PK 매칭으로 이전 선택을 복원
③ 신규 행(EDIT_STS:"I") 이 추가될 때 마지막 행으로 자동 스크롤

핵심 입력

11-4. 왜 이런 분해인가

  • 본체는 props 분배만 한다. 모든 결정은 hook 안에서 끝나므로, 본체 JSX 를 읽으면 "어떤 hook 이 무엇을 결정하는지" 만 보면 됩니다.
  • reference 안정성이 cellEditor 생존에 결정적. ag-grid 는 prop reference 가 바뀌면 내부 상태를 reset 합니다. 그래서 useGridProps 가 단독 hook 으로 떨어져 있고, useActivePreset 안에서도 setRowData 를 일부 deps 에서 일부러 제외(Proxy 가 매번 새 ref 반환)하는 식의 미묘한 처리가 들어 있습니다.
  • "active* 도출" 이 가장 복잡. tab/preset 활성화 분기는 columnDefs/rowData/actions/handler/render 등 거의 모든 것을 흔듭니다. 가장 위에서 한 번 결정하고, 아래 hook 들은 그 결과를 단순 소비합니다.
  • effect 와 callback 을 섞지 않음. useRowLifecycle 은 effect 전용, useGridHandlers 는 useCallback 전용. 한 hook 안에 둘이 섞이면 dep 추적이 어려워져 무한 루프/리셋 버그가 생기기 쉬움.
